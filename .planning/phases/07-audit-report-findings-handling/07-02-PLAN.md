<objective>
Fix `gru-triton-n20` (the shared-`QuantizerConfig` silent-correctness bug), absorb the Phase 4 strict-test re-baseline it triggers, and introduce the `divergence` pytest marker (D-05) — all in ONE plan because the n20 re-baseline and the marker application both write all 4 `tests/test_triton_*_strict.py` files plus `pyproject.toml`. Concurrent plans here would re-create the exact `gru-triton-u00` parallel-write race.

Purpose: `n20` is a genuine silent-correctness bug — `freeze_all` mutates a shared config to `mode='frozen'`, so the second sibling quantizer skips its stat-copy and stays at `scale=1.0`. The audit must not ship it. The fix changes quantizer scales, breaking ~18+ Phase 4 strict tests whose bit-identity contract depended on the buggy shared `scale=1.0`; re-baselining/marking those is in-scope Phase 7 work (D-07), not a regression. The `divergence` marker (D-05) operationalizes ROADMAP criterion #3's honest green gate.
Output: deepcopy fix in `make_quantizer`, extended CAL-02 failing test, registered `divergence` marker, divergence-marked strict cases, timestamped pytest-output artifact.

HARDWARE: this plan REQUIRES a CUDA + Triton host (RTX 2000 Ada per `04-DISPOSITION.md`) — the strict-test re-baseline numbers and the marker green-gate verification cannot be produced on CPU. The n20 deepcopy fix itself + the CAL-02 failing test are CPU-runnable, but the re-baseline discovery step is GPU-only.

WAVE-2 GPU-HOST GATE (WARNING 5): Task 2 (n20 strict-test re-baseline + marker verification) is GPU-only — its only `<verify>` is a CUDA gate. **Wave 2 execution BLOCKS on CUDA+Triton host availability.** If executed on a CPU-only host, Task 2's CUDA-gated verifications skip and Task 2 is NOT complete — it must be re-run and verified on a GPU host before Wave 2 is marked done. Critically: **Waves 3 and 4 (bd-issue closure, AUDIT-REPORT authoring) MUST NOT be marked complete on the back of an unverified Wave 2.** Wave 3's bd resolution notes and Wave 4's AUDIT-REPORT both report the FINAL post-fix strict-test state — that state is only real once Task 2's `-m "not divergence"` fast+slow gates have actually passed on a CUDA host. A skipped (not passed) Wave-2 CUDA gate is a hard blocker on Waves 3-4, not a soft warning.
